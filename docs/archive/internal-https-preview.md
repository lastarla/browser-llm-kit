# 内网 HTTPS 预览

目标：让 `Web LLM` 在内网环境下也能运行新的 `OPFS + Worker` 本地模型链路，并真实测试 1.9G 模型下载、安装和复用体验。

## 示例机器信息

- 局域网 IP：`192.168.1.20`
- Bonjour 主机名：`dev-machine.local`
- Node 服务地址：`http://192.168.1.20:3001/`

## 限制

- `http://192.168.1.20:3001/` 只能看页面，不能稳定测试新的浏览器端模型安装链路
- 原因是浏览器不会把局域网 IP 的纯 HTTP 页面当作安全上下文
- 真实 Web LLM 测试必须走 HTTPS 或本机 `localhost`，否则 `Worker + OPFS` 相关能力不完整

## 1. 安装 mkcert

```bash
brew install mkcert
mkcert -install
```

## 2. 生成证书

在仓库根目录执行：

```bash
mkdir -p .local/certs
mkcert \
  -cert-file .local/certs/browser-llm-kit-preview.pem \
  -key-file .local/certs/browser-llm-kit-preview-key.pem \
  browser-llm-kit.test \
  192.168.1.20 \
  dev-machine.local \
  localhost \
  127.0.0.1
```

## 3. 给测试机配置 hosts（可选）

由于证书里已经包含 `192.168.1.20`，直接访问 IP 也可以，不一定非要改 hosts。

如果你想给体验链接起一个固定名字，再让每台测试机加一条 hosts：

```text
192.168.1.20 browser-llm-kit.test
```

macOS/Linux：`/etc/hosts`

Windows：`C:\Windows\System32\drivers\etc\hosts`

## 4. 启动业务服务

```bash
npm start
```

## 5. 直接用 Node 启动原生 HTTPS

```bash
npm run start:https
```

## 6. 最终访问地址

最省事的地址是：

```text
https://192.168.1.20:3443/
```

如果测试机已经配置了 hosts，并且想用固定名字，也可以用：

```text
https://browser-llm-kit.test:3443/
```

如果只在当前机器自己测试，也可以直接用：

```text
https://localhost:3443/
```

## 7. 可选：继续用 Caddy

如果你更习惯反向代理，仓库里仍然保留了 [`examples/meeting-notes-demo/Caddyfile.internal`](../../examples/meeting-notes-demo/Caddyfile.internal)。

## 8. 证书信任说明

`mkcert` 生成的证书默认只对安装了对应根证书的机器受信任。

如果你要让“别人”也无警告访问，有两种办法：

1. 给每台测试机安装同一个 mkcert 根证书
2. 改用公司内网 CA / 已受信任的企业证书体系

如果别人没有信任根证书，页面仍然能打开，但浏览器会报证书不受信任，`Web LLM` 链路也不应作为有效测试结果。

### macOS 安装 `rootCA.pem` 详细步骤

建议把下面这段单独发给测试同事，基本可以按图索骥操作：

1. 先退出正在打开的 Chrome、Edge、Safari 等浏览器，避免证书已经装好但浏览器还没刷新信任状态
2. 在 Finder 里找到我发的 `rootCA.pem` 文件，建议先保存到桌面，方便后面查找
3. 双击 `rootCA.pem`
4. 如果系统弹出“要将证书添加到哪个钥匙串”，选择“系统”
5. 点“添加”或“好”，这一步通常会要求输入 macOS 登录密码，或者用 Touch ID 确认
6. 打开“钥匙串访问”
7. 左侧选择“系统”，上方或左侧分类里选择“证书”
8. 在右上角搜索框输入 `rootCA`，如果搜不到，也可以搜索 `mkcert`，找到刚导入的那张证书
9. 双击这张证书，打开证书详情窗口
10. 展开“信任”
11. 把“使用此证书时”从“使用系统默认值”改成“始终信任”
12. 关闭证书详情窗口，系统通常会再次要求输入 macOS 登录密码，这是正常现象，输完即可保存
13. 回到“钥匙串访问”，确认这张证书还在“系统”钥匙串里，并且没有明显的红色信任警告
14. 重新打开浏览器，访问 `https://192.168.1.20:3443/`

如果双击 `rootCA.pem` 之后没有打开“钥匙串访问”，而是被别的程序打开了：

1. 打开“钥匙串访问”
2. 在菜单栏选择“文件” -> “导入项目...”
3. 选择 `rootCA.pem`
4. 导入到“系统”钥匙串
5. 然后重复上面的“信任”设置步骤

安装完成后，可以这样快速确认是否成功：

1. 用浏览器打开 `https://192.168.1.20:3443/`
2. 地址栏不应该再出现明显的“不安全”提示
3. 点击页面里的 `Web LLM` 功能时，不应该再因为 HTTPS / 证书问题被浏览器拦住

如果仍然提示证书不受信任，优先按下面顺序排查：

1. 确认证书装在“系统”钥匙串，不是“登录”钥匙串
2. 确认证书详情里的“使用此证书时”已经改成“始终信任”
3. 完全退出浏览器后重新打开，再访问一次
4. 还不行的话，重启一次 macOS 再试
5. 如果公司设备没有管理员权限，可能需要 IT 或设备管理员协助输入密码完成导入

## 9. 发给测试同事的通知模板

可以直接把下面这段发给测试同事：

```text
请大家帮忙体验一下浏览器运行 gemma4 模型，这次主要测试新的 Web LLM 本地模型链路，关注首次下载、安装和二次复用体验，模型文件约 1.9G。

体验地址：
https://192.168.1.20:3443/

第一次访问前，需要先信任测试证书。
我会把根证书文件单独发给大家，文件名是 rootCA.pem。

macOS：
1. 先退出 Chrome、Safari、Edge 等浏览器
2. 双击打开 rootCA.pem
3. 如果系统询问“要添加到哪个钥匙串”，请选择“系统”
4. 点“添加”后，按提示输入 macOS 登录密码或用 Touch ID 确认
5. 打开“钥匙串访问”
6. 在左侧选择“系统”，再点“证书”
7. 在右上角搜索 rootCA 或 mkcert，找到刚导入的证书
8. 双击这张证书，展开“信任”
9. 将“使用此证书时”改成“始终信任”
10. 关闭窗口保存，系统如果再次要求输入密码，正常输入即可
11. 重新打开浏览器，再访问体验地址
12. 如果地址栏不再提示明显“不安全”，一般就说明证书已安装成功

如果双击 rootCA.pem 没反应，或者没有自动进入“钥匙串访问”：
1. 手动打开“钥匙串访问”
2. 菜单栏点“文件” -> “导入项目...”
3. 选择 rootCA.pem
4. 导入到“系统”钥匙串
5. 再按上面的第 6 到第 12 步继续操作

Windows：
1. 先退出 Chrome、Edge 等浏览器
2. 双击打开 rootCA.pem
3. 点击“打开”或“打开方式”，进入证书查看窗口后，点“安装证书”
4. 如果系统询问安装给谁，优先选择“本地计算机”
5. 点“下一步”
6. 选择“将所有的证书都放入下列存储”
7. 点“浏览”
8. 选择“受信任的根证书颁发机构”
9. 点“确定”后继续“下一步”
10. 点“完成”
11. 如果弹出安全提示，选择“是”
12. 重新打开浏览器，再访问体验地址
13. 如果地址栏不再提示证书不安全，一般就说明安装成功

如果没有“本地计算机”权限，或者系统提示权限不足：
1. 可以先尝试选择“当前用户”继续安装
2. 如果公司电脑被策略限制，可能需要 IT 或设备管理员协助安装

装完证书后：
1. 重新打开浏览器
2. 访问 https://192.168.1.20:3443/
3. 点击左侧任意样本的 Web LLM
4. 观察首次下载、安装进度、首次可用时间和是否成功进入可用状态

重点帮我关注：
- 页面是否还能提示“不安全”
- 点击 Web LLM 后是否有反应
- 首次下载并安装 1.9G 模型是否顺利
- 大概耗时多久
- 中途有没有卡住、报错、刷新后恢复不了
- 第二次再点时是否明显变快，是否直接复用本地安装结果

如果已经完成安装，后续再次点击应优先复用本地安装结果，而不是重新走完整下载。

如果遇到问题，请尽量把下面信息一起发我：
- 电脑系统和浏览器版本
- 访问时间
- 报错截图
- 是否第一次访问
- 是否已经安装并信任 rootCA.pem
```
